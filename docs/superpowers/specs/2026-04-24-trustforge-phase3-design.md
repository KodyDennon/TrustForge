# TrustForge Phase 3 — Session Protocol Prototype Design

**Date:** 2026-04-24
**Status:** Draft, approved for implementation planning
**Scope:** Roadmap Phase 3 — binary framing, handshake skeleton, mutual authentication, session metadata, rekey hooks, WebSocket-carried mode.

## 1. Purpose

Two TrustForge actors must be able to:
- Open a live mutually-authenticated session.
- Exchange integrity-protected, encrypted frames.
- Rekey mid-session.
- Carry the session over a WebSocket so it works in any networked environment.

The result is a working prototype, not a TLS replacement. It is honest about what it does (X25519 ephemeral + identity signature + AEAD), composes only reviewed primitives, and is small enough to read in one sitting.

## 2. Non-goals

- **No custom crypto primitives or modes.** Build only with X25519 (`@noble/curves` / `x25519-dalek`), HKDF-SHA256 (`@noble/hashes` / `hkdf` crate), ChaCha20-Poly1305-IETF (`@noble/ciphers` / `chacha20poly1305` crate), and the existing ed25519.
- **Not Noise framework.** A future spec may swap to Noise XX; the API is shaped so that swap is local. The prototype's protocol is a conventional ECDHE-with-identity-signature exchange, no novel construction.
- **No PSK, no 0-RTT, no resumption, no replay-window negotiation.** Strict monotonic sequence numbers and a single sliding window of size 1.
- **No PQ key exchange.** Hybrid KEM is Phase 6+.
- **No session migration across transports**, no relay forwarding semantics. One transport per session.

## 3. Wire protocol

Each side identifies as a TrustForge actor with a 32-byte ed25519 identity public key (already serialised in `actor-identity.schema.json`). Each side also has an ephemeral X25519 keypair generated per handshake.

### 3.1 Handshake (3 messages)

```
Init ─┐ HelloI         { version, suite, session_id, peer_hint, eph_pub_I }
      ├──────────────────────────────────────────────────────────────►
      │
Init  │           HelloR { eph_pub_R, ident_pub_R, sig_R(transcript_so_far) }
      ◄──────────────────────────────────────────────────────────────┐ Resp
      │
Init ─┐ Auth          { ident_pub_I, sig_I(transcript_so_far) }
      ├──────────────────────────────────────────────────────────────►
      │
                       (both sides derive session keys, ready to send AEAD frames)
```

- `version` = `0` (prototype).
- `suite` = `"x25519-hkdf-sha256-chacha20poly1305-ed25519"` (only suite for now).
- `session_id` = 16 random bytes generated by the initiator.
- `peer_hint` = the actor URI the initiator believes the responder is, for routing — verified after the handshake.
- `transcript_so_far` is the concatenation of canonical-JSON forms of all preceding handshake messages.

After exchanging keys:
1. Both compute `shared = X25519(eph_priv_self, eph_pub_peer)`.
2. Both derive keys: `(send_key_I, send_key_R, mac_salt) = HKDF-SHA256(shared, salt = session_id, info = "tf-session/v0/keys" || transcript_hash)`.
3. Both store: own actor URI, peer actor URI (verified from `ident_pub_*` against the actor-identity record), keys, generation = 0, send seq = 0, recv seq = 0.

The signature on each side covers `sha256(transcript)`. Either side rejects the handshake if:
- `version` or `suite` mismatches.
- The peer's `ident_pub_*` does not verify against the supplied signature on the transcript.
- (Out of scope: the peer's `ident_pub_*` does not match a known actor-identity. The Phase 3 demo trusts the bound public key; Phase 4 wires this up against the actor-identity registry.)

### 3.2 Frame format

After handshake, every frame on the wire is:

```
+---+---+---+---+---+---+---+---+
| length (u32 BE)               |   total bytes that follow
+---+---+---+---+---+---+---+---+
| seq (u64 BE)                  |   monotonic per direction
+---+---+---+---+---+---+---+---+
| ciphertext + tag (16 bytes)   |   ChaCha20-Poly1305 IETF
+-------------------------------+
```

- `nonce` = 4 zero bytes || `seq` as u64 BE (12 bytes total) — the ChaCha20-Poly1305-IETF nonce.
- `aad` = `length || seq` (the framing header). Authenticates that the framing header was not tampered.
- `seq` rolls over → fail closed: re-establish the session.
- A receiver enforces `seq == expected_seq`; out-of-order frames close the session.

### 3.3 Frame types (inside the cleartext)

The plaintext of every data frame is a canonical-JSON object with one of:

```ts
type SessionFrame =
  | { kind: "data";  payload: unknown }
  | { kind: "rekey-req"; eph_pub: base64 }
  | { kind: "rekey-ack"; eph_pub: base64 }
  | { kind: "close"; reason?: string }
  | { kind: "ping"; nonce: base64 }
  | { kind: "pong"; nonce: base64 }
```

`data` carries opaque application bytes (e.g., a serialized ProofRPC call later). Phase 3 is transport-only and does not interpret the payload.

### 3.4 Rekey

Rekey is in-band:

1. Either side sends `{ kind: "rekey-req", eph_pub: <new ephemeral X25519 pubkey> }` over the current session.
2. Peer responds with `{ kind: "rekey-ack", eph_pub: <its new ephemeral X25519 pubkey> }`.
3. Both derive new keys: `(send_key, recv_key, mac_salt) = HKDF(shared_2, salt = session_id, info = "tf-session/v0/keys/g{N+1}" || prev_keys_hash)` where `prev_keys_hash = sha256(prev_send_key || prev_recv_key)`.
4. Both reset `send_seq = 0`, `recv_seq = 0`, `generation += 1`.
5. The first frame after rekey uses the new keys.

A side that has sent `rekey-req` but not yet received `rekey-ack` MUST NOT send `data` frames; it may only send another `rekey-req` (idempotent retransmit) or `close`.

### 3.5 Session metadata

Both sides keep:

```ts
type SessionState = {
  session_id: Uint8Array;         // 16 bytes
  self_actor: string;
  peer_actor: string;
  generation: number;             // increments on rekey
  send_seq: bigint;
  recv_seq: bigint;
  send_key: Uint8Array;           // 32 bytes
  recv_key: Uint8Array;           // 32 bytes
  closed: boolean;
};
```

This is the structured representation; serialization is for tests and proof-event bookkeeping (a future `session.established` proof event will reference `session_id`, `generation`, and the peer actor).

## 4. WebSocket carrier

A `tools/tf-session` package binds the handshake + frame protocol to a WebSocket transport:

- Server: a Bun WebSocket server that accepts incoming connections, runs the responder side of the handshake, then exposes typed `send(SessionFrame)` / `onReceive((SessionFrame) => void)` callbacks.
- Client: a Bun client that does the same on the initiator side.

There is one frame per WebSocket binary message — no fragmentation logic. WebSocket already does framing.

## 6. Phases

1. **R1** — crypto suite extension: X25519 keygen + ECDH, HKDF-SHA256, ChaCha20-Poly1305-IETF AEAD. Both languages, with conformance/session-vectors.yaml pinning every primitive's output.
2. **R2** — handshake state machine: Initiator + Responder driving the 3-message exchange. End-to-end: in-memory pipe between TS and Rust state machines via the same vectors.
3. **R3** — session state + frame encrypt/decrypt: implement `send(payload)` and `recv(bytes)` that use seq numbers and AEAD framing. Vector-pin the byte output of one frame for parity.
4. **R4** — rekey: in-band rekey-req / rekey-ack flow with key derivation + seq reset. Test rekey both directions.
5. **R5** — WebSocket carrier `tools/tf-session`: server + client + e2e test (both halves run in the same Bun process).
6. **R6** — CI updates: new vector tests + e2e wired into the workflow.

## 7. Done criteria

- Both runtimes derive the same X25519 shared secret, HKDF output, and AEAD ciphertext for shared inputs.
- A TS initiator handshakes with a Rust responder (via the in-memory adapter) and both sides exchange at least one `data` frame round-trip.
- The same is true with TS responder + Rust initiator.
- Rekey produces fresh key material with `generation > 0`; out-of-order frames after rekey are rejected.
- The Bun WebSocket demo runs without the test suite cheating (real TCP loopback through Bun.serve).

## 8. Risks

- **ChaCha20-Poly1305 nonce reuse**: monotonic seq + key-rotation-on-rekey makes this safe but it must be tested. A test deliberately tries to send seq=N twice and asserts the receiver rejects the second.
- **Identity-signature transcript binding**: the transcript hash includes every message before the signature is computed. If we get the order wrong, MITM is possible. The vector file pins the exact transcript bytes so any drift is caught.
- **Bun WebSocket binary path**: Bun's WebSocket lets you opt into binary frames. Confirm both sides produce `ArrayBuffer`/`Uint8Array` not strings.
- **No replay protection across reconnects**: out of scope for the prototype; documented.
