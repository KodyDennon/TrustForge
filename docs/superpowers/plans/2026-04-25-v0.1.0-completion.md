# v0.1.0 Completion — End-to-End Implementation Plan

**Goal:** Close every finding from the 4 audits (spec-drift, bug-detector, test-coverage, architecture) so v0.1.0 actually delivers what the README and CHANGELOG claim: every spec / profile / bridge / DECISIONS.md cross-cutting requirement working in BOTH TypeScript and Rust, gated by a real conformance suite.

**Architecture principles (apply across every phase):**
1. **Spec ↔ implementation moves together.** When code lands, the spec is updated to match (or vice versa).
2. **Every TS module has a Rust counterpart and a parity vector.** No more "TS-only" features in v0.1.0.
3. **Every advertised feature has a runtime test that exercises it through the real path** (not the unit-test stub).
4. **The conformance gate refuses to silent-pass.** Every category must execute work, surface failures, and feed CI.
5. **No custom cryptography.** Every crypto primitive comes from a reviewed library; if a primitive isn't available in both TS and Rust, the feature waits.
6. **Fail-closed defaults.** Optional verification arguments are removed; callers must opt out explicitly.

**Tech stack additions this plan introduces:**
- Rust `fips204` crate (NIST FIPS 204 ml-dsa-44/65/87) for hybrid PQ.
- Rust `wasmtime` for the WASM plugin runtime.
- TS Web Crypto `crypto.subtle` (Bun-native) for RSA verification, replacing the hand-rolled implementation.
- TS `@peculiar/x509` (already a dep) wired into real WebAuthn cert chain validation.
- Rust `x509-parser` extended for chain validation, OCSP, CRL.
- TS `@cedar-policy/cedar-wasm` + Rust `cedar-policy` crate for Cedar policy.
- TS `@open-policy-agent/opa-wasm` + Rust `regorus` crate for Rego policy.
- Native plugin sandbox: child-process spawn with Linux `seccomp-bpf` (via `seccompiler` crate behind a small Rust helper) and macOS `sandbox-exec` profile; Windows path TBD (refused on Windows for v0.1.0).
- CBOR (RFC 8949) deterministic encoding (`cbor-x` TS, `ciborium` Rust — both already deps) for `.tfbundle` and `.tfpkt`.

**Locked decisions (from user clarifications 2026-04-25):**
1. **Native plugin sandbox:** real isolation via child-process + seccomp (Linux) / sandbox-exec (macOS); refuse on Windows. WASM remains the cross-platform path.
2. **Cedar/Rego:** real implementations in both TS and Rust via the wasm/native crates above.
3. **Wire format:** clean break for v0.1.0 — no compat shims for 0.0.0.
4. **Hybrid-PQ composition:** parallel — `{ed25519_signature, mldsa_signature}` both over the same transcript hash, both must verify. Matches NIST CNSA 2.0 and IETF hybrid-signature drafts.
5. **Branch strategy:** directly on main, sprint-style commits (matches Sprints 6/7 pattern).
6. **ProofRPC method kinds:** all 10 implemented distinctly (subscribe, server-streaming, client-streaming, bidi-streaming, command-channel, bulk-transfer, telemetry, remote-shell, agent-session, unary).
7. **Binary format:** CBOR (RFC 8949) deterministic encoding for `.tfbundle` and `.tfpkt`.
8. **Bridge spec depth (B16):** full normative 200-300 line specs each.

**Estimated commit count:** ~18 (one per batch, all green at handoff).

## Phase 3 — Sandboxing

### Batch B5 — Plugin sandbox: real isolation (TS + Rust + WASM)

**Closes:** BUG-018 (Worker not actually sandboxed), BUG-019 (WASM imports leak), BUG-021 (revocation only checked at load), FIND-042 (Rust skips WASM), FIND-043 (host bag exposed).

**Approach (per locked decision):** native plugins run in a **separate child process** isolated via OS primitives. WASM plugins run in `wasmtime` (Rust) and `WebAssembly` (TS) with imports gated through AgentGuard.

**Files:**

*TS native sandbox*
- Replace: `tools/tf-types-ts/src/core/plugin.ts:179-262` (`loadNativeInWorker`) with `loadNativeInChildProcess`:
  - Spawns a Bun subprocess via `Bun.spawn(["bun", "run", helperPath], {stdio:[pipe,pipe,pipe], env: minimalEnv})`.
  - On Linux: helper applies a seccomp-bpf filter (via the `seccompiler` Rust binary `tf-sandbox-helper-linux` shipped under `tools/tf-sandbox-helper/`) before `await import(entryPath)`. Filter denies: `openat`, `connect`, `bind`, `socket`, `execve`, `clone`, `fork`, `mmap`/`mprotect` with `PROT_EXEC`, plus the obvious filesystem syscalls. Allows: `read`, `write`, `mmap` without exec, `futex`, `clock_gettime`.
  - On macOS: helper invokes `sandbox-exec -f tools/tf-sandbox-helper/profile.sb` with a profile that denies `file*`, `network*`, `process-fork`, `process-exec`, allows `mach-lookup` only for the parent IPC port.
  - On Windows: refuses to load native plugins; logs an explicit error directing users to WASM. `unsafe_allow_native_plugins: true` is honored only on Windows for opt-out.
  - Communication via stdin/stdout JSON-line protocol: parent sends `{call_id, method, args}`, child responds `{call_id, result | error}`. Host imports are not exposed to the child; the child can only call back into the parent via the IPC channel, and each callback is a guard-checked capability.
- Add: `tools/tf-sandbox-helper/` — small Bun script + native helper binaries (one per platform).
  - `tools/tf-sandbox-helper/index.ts` — Bun child-process entrypoint: applies sandbox, then loads the plugin, then handles JSON-line IPC.
  - `tools/tf-sandbox-helper/profile.sb` — macOS sandbox profile (deny default, allow mach-lookup parent).
  - `crates/tf-sandbox-helper-linux/src/main.rs` — Linux seccomp helper, compiled as a small binary that the Bun child execs into before loading user code.

*WASM gating (TS)*
- Modify: `tools/tf-types-ts/src/core/plugin.ts:117-128, 156-172` — wrap every host import in a guard check: `(...args) => { if (!guard.check({actor: manifest.actor_id, action: "wasm.import." + importName})) throw new PluginError("import refused"); return host[importName](...args); }`.

*Runtime revocation re-check*
- Modify: `tools/tf-types-ts/src/core/plugin.ts:67-150` — `PluginRegistry` constructor takes a `RevocationIndex` reference; on every plugin handler invocation (in `registerOn`'s wrapper), `revocations.isRevoked({id: actor_id, kind: "actor"}, now)` is checked; revoked actors → `permission_denied`.

*Rust WASM*
- Add: `crates/tf-types/Cargo.toml` — `wasmtime = "25"`.
- Add: `crates/tf-types/src/plugin_wasm.rs` — `WasmPluginRuntime` with `Linker` exposing only manifest-declared imports, each wrapped with the same guard-check shim. Mirror TS API.
- Modify: `crates/tf-types/src/plugin.rs:9-10` — remove the "out of scope" comment; route to `plugin_wasm`.

*Native sandbox in Rust*
- Add: `crates/tf-types/src/plugin_native.rs` — for parity, Rust can spawn a child Rust binary running the plugin under the same seccomp/sandbox-exec profile. Cleaner than the TS path because the parent owns the helper binary directly.

*Example*
- Add: `plugins/tf-plugin-example-wasm/` — minimal WASM plugin demonstrating the capability sandbox (one allowed import, one disallowed import that fails to instantiate).
- Add: `plugins/tf-plugin-example-native/` — minimal Bun-native plugin demonstrating the sandbox refuses filesystem access.

## Sequencing rationale

- **B1 first** because everything else (guard tests, federation, daemon e2e) needs the actor model working.
- **B2 immediately after** because every signature-bearing test needs canonical-JSON parity to pass cross-language.
- **B3 third** because crypto correctness underlies every signed object (proofs, bundles, attestations, packets, plugin manifests).
- **B4 fourth** because the daemon is the integration point — once foundations are right, daemon hardening can land without rework.
- **B5–B9** are independent of each other; could parallelize, but staying serial keeps each commit small.
- **B10–B11 cluster** so the conformance gate becomes real before the parity work in B12–B15 lands; B12–B15 each then add to the now-real gate.
- **B16 late** so docs/specs match the now-stable implementation.
- **B17 reaches deep into every prior batch** — easier to write the tests after the implementation has settled.
- **B18 last** so CHANGELOG describes what actually shipped.
